# Metaflow UI Frontend: Live DAG, Run Comparison, Theming

Metaflow’s React UI currently fetches flow/run data from a service and shows a static DAG. The new features will require **real-time updates**, a **diff view**, and theming changes. In practice, the `metaflow-ui` frontend must be updated to consume new backend endpoints (or WebSocket streams) provided by the `metaflow` CLI/server. For example, the UI can poll or open a WebSocket to receive step status and log updates as they happen【87†L69-L74】. The GSoC goals explicitly call for *“Live DAG visualization”* where steps light up in real-time with streaming logs【91†L273-L279】, a *“Run comparison view”* showing side-by-side parameter/artifact diffs【91†L273-L279】, and a *“Dark mode theme”* toggle【91†L287-L290】. Below is how these map to implementation:

- **Live-updating DAG**: We will replace the static graph with a dynamic renderer (e.g. using D3.js or a library like [React Flow](https://reactflow.dev/) for DAGs). The frontend will subscribe to updates via WebSockets (or frequent API polling). Each time a step completes, the backend pushes a JSON update; the React app updates the node’s color/status and appends new log text. Using WebSockets keeps a persistent two-way channel, which “enabl[es] instant data updates with minimal overhead,” making it ideal for real-time apps【87†L69-L74】. In code, a WebSocket listener (or `EventSource`) feeds incoming step data into state, and the DAG component re-renders accordingly. The UI should also display a streaming log viewer (e.g. appending lines to a console pane) as logs arrive. This satisfies the “live DAG visualization” requirement【91†L273-L279】.

- **Run Comparison View**: The UI will add a new page or panel that lets the user select two runs of the same flow. It then fetches both runs’ details (parameters, artifacts, metrics) via the backend API. In React, we can show this as two side-by-side tables or cards, highlighting differences. For example, parameter values that differ can be colored, and artifact sizes or metrics can show deltas. Libraries like `react-diff-view` can render side-by-side diffs of JSON if needed. This directly implements the “side-by-side diff” deliverable【91†L273-L279】. (No new real-time technique is needed here; it’s a static comparison after both runs complete, but the frontend must organize and highlight the differences clearly.)

- **Dark Mode & Theming**: A theme toggle will let users switch between light and dark. In practice, we’d use CSS variables or a CSS framework’s dark mode support. For example, with Tailwind CSS you can use the `dark:` variant to style elements based on a parent `.dark` class【94†L272-L274】. Clicking the toggle would add/remove `class="dark"` on the HTML or body element (or use `prefers-color-scheme`). We should also update our components’ styles (colors, icons) to ensure readability in dark mode. As UI guides note, dark mode is now expected by users (it “reduce[s] eye strain” and adds a polished look【89†L82-L85】). Implementing this might use a ThemeContext in React and persist the choice in local storage, similar to many modern apps【89†L82-L85】【94†L272-L274】.

- **Integration between repos**: The `metaflow` (Python) repo must expose any new data needed by the frontend. For live updates, the backend could use FastAPI with WebSocket endpoints or Server-Sent Events to push step events. For comparisons, it should offer endpoints like `/runs/{id}/compare/{otherId}` that return both runs’ data. The `metaflow-ui` frontend will call these endpoints (or open the socket) to retrieve JSON. The existing UI code will be refactored to plug in the new DAG component and comparison component; it already handles flows and runs via API calls, so it will now target the local backend instead of a remote service. 

In summary, the frontend work involves adding dynamic visualization components (e.g. a React/D3 DAG that updates from live data)【91†L273-L279】, a diff-view component for runs【91†L273-L279】, and global theme support (dark mode)【91†L279-L280】【94†L272-L274】. These features, combined with real-time data delivery via WebSockets【87†L69-L74】, will meet the project’s goals of live DAG updates, run diffing, and a modern dark-themed UI as specified in the GSoC proposal【91†L273-L279】【91†L287-L290】.
